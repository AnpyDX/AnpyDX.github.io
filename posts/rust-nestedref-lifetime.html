Rust 嵌套引用生命周期的推断 ​

#Rust

问题 ​

为了理解为什么需要推断嵌套引用的生命周期，这里以光追渲染器中的一段代码为例。

rust
/// 对 `屏幕` 的抽象
struct Screen {
    buffer: Vec<u32>
}

struct TaskIter<'a> {
    owner: &'a mut Screen,
    counter: usize,
    ending: usize
}

impl Screen {
    /// 将屏幕像素集合转换为迭代器，用于遍历
    pub fn iter<'a>(&'a mut self) -> TaskIter<'a> {
        let len = self.buffer.len();
        TaskIter { owner: self, counter: 0, ending: len }
    }
}

为了能够实现目标行为（遍历操作），尝试为 TaskIter 实现 Iterator trait。

rust
impl<'a> Iterator for TaskIter<'a> {
    type Item = &'a mut u32;

    fn next(&mut self) -> Option<Self::Item> {
        // 方便起见，这里简化了具体算法
        Some(&mut self.owner.buffer[0])
    }
}
rust
let mut screen = Screen { buffer: vec![0; 10] };

for task in screen.iter() {
    // do sth with task.pixel
    todo!()
}

但 rustc 却给出了下面的生命周期报错：

error: lifetime may not live long enough
  --> src/main.rs:23:9
   |
19 | impl<'a> Iterator for TaskIter<'a> {
   |      -- lifetime `'a` defined here
...
22 |     fn next(&mut self) -> Option<Self::Item> {
   |             - let's call the lifetime of this reference `'1`
23 |         Some(&mut self.owner.buffer[0])
   |         ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ method was supposed to return data with lifetime `'a` but it is returning data with lifetime `'1`

分析 ​

通过编译器给出的报错信息可知，生命周期问题出现在 fn next(&mut self) 上，具体错误为：pixel 的生命周期可能不够长。

为了方便理解，这里我们对 trait 实现部分进一步简化，并手动标注所有引用生命周期，如下：

rust
// notice: 'a >= 'b
impl<'b, 'a: 'b> Iterator for TaskIter<'a> {
    fn next(&'b mut self) -> Option<&'a mut u32> {
        Some(&mut self.owner.buffer[0])
    }
}

到这里问题就不言而喻了：我们在尝试通过一个 &'b mut TaskIter<'a> 拿到 &'a mut u32。或者说我们正在通过 &'b mut &'a mut T 拿到 &'a mut T。

为什么上述情况不被 rustc 允许呢？这里就要引入文章要探讨的核心规则：

  1. &'b mut &'a mut T 可以得到 &'b mut T，但不能得到 &'a mut T。
  2. &'b mut &'a T 可以得到 &'b T，也可以得到 &'a T。

可以试着编译下方的代码来验证上述规则。

rust
struct Buffer {
    data: u32
}

struct MutWrapper<'a> {
    owner: &'a mut Buffer
}

struct Wrapper<'a> {
    owner: &'a Buffer
}

// ok
fn try_convert_0<'b, 'a: 'b>(r: &'b mut MutWrapper<'a>) -> &'b mut u32 {
    &mut r.owner.data
}

// error
fn try_convert_1<'b, 'a: 'b>(r: &'b mut MutWrapper<'a>) -> &'a mut u32 {
    &mut r.owner.data
}

// ok
fn try_convert_2<'b, 'a: 'b>(r: &'b mut Wrapper<'a>) -> &'a u32 {
    &r.owner.data
}

// ok
fn try_convert_3<'b, 'a: 'b>(r: &'b mut Wrapper<'a>) -> &'b u32 {
    &r.owner.data
}

fn main() {}

为什么 mut 能影响到生命周期呢？这与核心库中 &T 和 &mut T 的 Deref 和 DerefMut 实现有关。

为了实现嵌套引用的自动解引用功能，Rust 为 &T 和 &mut T 选择性地实现了Deref 和 DerefMut，核心代码如下：

Rust
// impl Deref for &T
fn deref(&'b &'a T) -> &'b T

// impl Deref for &mut T
fn deref(&'b &'a mut T) -> &'b T

// impl !DerefMut for &T
unimplemented!()

// impl DerefMut for &mut T
fn deref_mut(&'b mut &'a mut T) -> &'b mut T

通过分析 trait 实现，可以验证这几条规则的合理性：

  • &'b mut &'a T -> &'b &'a T -> &'b T
  • &'b mut &'a mut T -> &'b mut T
  • &'b mut &'a mut T -!-> &'a mut T

​

但是根据实现，&'b mut &'a T -> &'a T 理应也不成立，为什么实际却可行呢？

这是因为 Rust 还为 &T 特别地实现了 Clone 和 Copy traits。

rust
pub trait Clone: Sized {
    fn clone(self: &Self) -> Self;
}

pub trait Copy: Clone {}

根据 clone 的方法签名可知，当 &'b mut &'a T 匹配到 &self 时，返回的是Self = &'a T（注意：生命周期标识也是类型的一部分），也就是说返回的引用拥有内层的生命周期！

可变引用由于其独占性，没有实现这两个traits，自然也就无法实现这样的转换。

解决方案 ​

由于 rustc 的静态分析是极其保守的，许多能从“逻辑”上确保安全的代码也会被拒绝。

那我们不妨从目标行为出发，来分析一下实际使用时各引用的行为是否安全。

rust
let mut screen = Screen { ... };

for task in screen.iter() {
    task.pixel = 1; // do some caculation
}

通过分析可以发现，如果严格按照上述行为，代码实际上是合理的。

  1. pixel 的生命周期只有一个循环，可变引用的作用域并不重叠。并且由于 TaskIter 中索引计数器的使用，可变引用不会指向同一个对象。因此没有违反 alias模型。

  2. 每一次 next 调用并不会改变 screen 内部，不存在后续操作造成先前引用悬垂。

因此，在不破坏封装的情况下，我在此想到了两种解决方案。

  • RefCell 实现的内部可变性
  • 利用 unsafe 抹除生命周期标识

内部可变性 ​

核心是利用 &'b mut &'a T 可以导出 &'a T 来解决生命周期问题，RefCell 来解决可变引用问题。

注意引用可变类型的变化。

rust
use std::cell::{ RefCell, RefMut };

#[derive(Debug)]
struct Screen {
    buffer: Vec<RefCell<u32>> 
}

struct TaskIter<'a> {
    owner: &'a Screen, 
    counter: usize,
    ending: usize
}

impl Screen {
    pub fn iter<'a>(&'a self) -> TaskIter<'a> { 
        TaskIter {
            owner: self,
            counter: 0,
            ending: self.buffer.len()
        }
    }
}

impl<'a> Iterator for TaskIter<'a> {
    type Item = RefMut<'a, u32>; 

    fn next(&mut self) -> Option<Self::Item> {
        if self.counter == self.ending {
            return None;
        }

        self.counter += 1;
        Some(self.owner.buffer[self.counter - 1].borrow_mut()) 
    }
}

fn main() {
    let screen = Screen { buffer: vec![RefCell::new(0); 10] };

    let mut index = 0;
    for mut pixel in screen.iter() {
        *pixel = index;
        index += 1;
    }

    println!("{:#?}", screen);
}

抹除生命周期标识 ​

由于“裸指针转换出的引用生命周期是 无界的(unbound)”，因此通过 ref -> raw_ptr -> ref ，便可以抹去引用生命周期标识，从而逃避检查。

rust
#[derive(Debug)]
struct Screen {
    buffer: Vec<u32>
}

struct TaskIter<'a> {
    owner: &'a mut Screen,
    counter: usize,
    ending: usize
}

impl Screen {
    pub fn iter<'a>(&'a mut self) -> TaskIter<'a> {
        let len = self.buffer.len();
        TaskIter {
            owner: self,
            counter: 0,
            ending: len
        }
    }
}

impl<'a> Iterator for TaskIter<'a> {
    type Item = &'a mut u32;

    fn next(&mut self) -> Option<Self::Item> {
        if self.counter == self.ending {
            return None;
        }

        self.counter += 1;
        Some(unsafe { 
            (&mut self.owner.buffer[self.counter - 1] as *mut u32) 
            .as_mut() 
            .unwrap() 
        }) 
    }
}

fn main() {
    let mut screen = Screen { buffer: vec![0; 10] };

    let mut index = 0;
    for pixel in screen.iter() {
        *pixel = index;
        index += 1;
    }

    println!("{:#?}", screen);
}
